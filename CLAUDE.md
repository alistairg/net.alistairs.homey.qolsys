# Homey Driver: Qolsys IQ Panel (IQ Remote Protocol)

## Source Reference
Protocol reverse-engineered by Eric Hylands. Reference implementations:
- Python module (MIT): https://github.com/EHylands/QolsysController
- HA integration: https://github.com/EHylands/ha-qolsys-panel

Both are cloned in the project root (`QolsysController/`, `ha-qolsys-panel/`).

---

## Architecture

The Qolsys panel runs a **built-in MQTT broker** secured with mutual TLS (mTLS). This app connects to that broker as an MQTT client, impersonating a virtual **IQ Remote** secondary keypad. No external broker required.

### Two Phases
1. **Pairing** — one-time TLS certificate exchange (Homey acts as TLS *server*, panel connects to Homey)
2. **Operation** — ongoing MQTT over mTLS using stored certificates

### Shared Client Pattern
One `QolsysPanelClient` per panel, shared across all devices (same pattern as the AVPro app). Managed in `app.ts`, referenced by both drivers.

## Key Implementation Details

### TypeScript Imports
- **app.ts, driver.ts, device.ts**: `import Homey from 'homey'`
- **lib/*.ts files**: `import Homey from 'homey/lib/Homey'` (for the Homey instance type)

### PKI Storage
All PKI material stored in `homey.settings` with prefix `pki_` — **never** in `/userdata/` (publicly accessible via HTTP).

Keys: `pki_private_key`, `pki_certificate`, `pki_csr`, `pki_secure`, `pki_qolsys_ca`, `pki_mac_address`

PKI details (from `PkiManager.ts`):
- RSA 2048-bit, exponent 65537
- Subject: `C=US, ST=SanJose, L=, O=Qolsys Inc., CN=www.qolsys.com ` (trailing space on CN is intentional — matches Python source)
- Random MAC format: `f2:16:3e:xx:xx:xx` (4th byte 0x00-0x7F)

### Pairing (PairingServer.ts)
1. TLS server on random port 50000–55000
2. mDNS advertisement via `bonjour-service`: name `NsdPairService`, type `http`
3. Panel connects → `\x00\x11` + MAC exchange → CSR → receive .secure + .qolsys (terminated by `b"sent"`)
4. 5 minute timeout, cleanup on completion or error

### MQTT Connection (QolsysPanelClient.ts)
- Port 8883, `rejectUnauthorized: false`, `keepalive: 60`
- Client ID: `qolsys-controller-{formatted_mac}`
- Subscribe: `iq2meid`, `response_{formatted_mac}`
- Publish: `mastermeid`
- Post-connect sequence: `connect_v204` → `pingevent` → `pair_status_request` → `syncdatabase`
- Command correlation: UUID `requestID` in payload, response on `response_` topic
- Ping: `pingevent` every 600 seconds
- Reconnection: exponential backoff 5s → 10s → 20s → 40s → 80s → max 120s
- SSL errors are fatal (bad certs, re-pair required)

### Arm/Disarm Commands
Arm and disarm use IPC-wrapped commands (`ipcCall` eventName):
- `ipcServiceName`: `qinternalservice`
- `ipcInterfaceName`: `android.os.IQInternalService`
- `ipcTransactionID`: 7

Disarm logic varies by partition state:
- Alarm → `disarm_from_emergency`
- Exit delay → `disarm_from_openlearn_sensor`
- Armed → send `ui_delay` first, then `disarm_the_panel_from_entry_delay`

### Database (DatabaseParser.ts)
`fulldbdata` is an array of ContentProvider entries: `{ uri, resultSet[] }`.

Key URIs:
- `PartitionContentProvider/partition` — partition list
- `SensorContentProvider/sensor` — zone/sensor data (47 columns)
- `QolsysSettingsContentProvider/qolsyssettings` — system settings per partition
- `StateContentProvider/state` — runtime state (ALARM_STATE)

Live updates arrive as `dbChanged` events on `iq2meid`:
```json
{
  "eventName": "dbChanged",
  "dbOperation": "update",
  "uri": "content://...",
  "contentValues": { "field": "newValue" },
  "selectionArgs": ["zoneId", "partitionId"]
}
```

### Zone Type Mapping (ZoneTypes.ts)
| sensortype | Homey Capability |
|---|---|
| `Door_Window`, `Door_Window_M`, `Tilt` | `alarm_contact` |
| `Motion`, `Panel Motion`, `Occupancy Sensor` | `alarm_motion` |
| `SmokeDetector`, `Smoke_M` | `alarm_smoke` |
| `CODetector` | `alarm_co` |
| `Water` | `alarm_water` |
| Everything else | `alarm_generic` |

All zones also get `alarm_tamper` + `alarm_battery`. Keypad and Bluetooth types are excluded.

Active statuses: `Open`, `Active`, `Activated`, `Alarmed`, `Occupied`

### Partition State Mapping
| Panel Status | Homey `homealarm_state` |
|---|---|
| `ARM-AWAY` | `armed` |
| `ARM-STAY`, `ARM-NIGHT` | `partially_armed` |
| `DISARM` | `disarmed` |
| `*-EXIT-DELAY` | (no change — arming in progress) |

### Flow Cards
**Alarm Panel triggers:** `alarm_triggered` (tokens: partition_name, alarm_type)
**Alarm Panel actions:** `arm_away`, `arm_home`, `disarm`
**Zone Sensor triggers:** `zone_tampered`

Standard Homey contact/motion triggers work automatically via capabilities.

### Device Boot Reconnection
On Homey reboot, `AlarmPanelDevice.onInit()` checks for an existing client. If none, it fetches a fresh local IP via `homey.cloud.getLocalAddress()` and creates + connects a new client using the stored panel IP.

## Homey SDK Notes

- **No DiscoveryStrategy** — pairing is reversed (panel connects to Homey), so we use custom pair views
- **mDNS advertising** — not built into Homey SDK. Using `bonjour-service` npm package (dgram UDP multicast)
- **TLS server sockets** — work on Homey Pro (proven by MQTT Server app)
- **Settings prefix `zone_`** — reserved by Homey. Use `sensor_zone_id` instead
- **User code** — stored in `homey.settings` key `user_code`, entered during pairing

### IQ Remote Slots
- **IQ Panel 2/2+**: up to 3 virtual remotes
- **IQ Panel 4 / IQ Pro**: up to 8 virtual remotes

Lost certs = lost slot. Certs in `homey.settings` survive app updates but not app uninstall.
